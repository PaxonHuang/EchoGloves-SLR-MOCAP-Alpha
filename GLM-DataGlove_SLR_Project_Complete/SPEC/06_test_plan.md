## 1. 测试策略概览

### 1.1 测试金字塔

本项目的测试策略遵循经典的测试金字塔模型，将测试活动按照覆盖粒度和执行成本进行分层组织。金字塔底层为数量最多、执行速度最快的单元测试，负责验证各独立模块的功能正确性；中层为集成测试，验证模块间接口和数据流的正确性；顶层为数量较少但覆盖面最广的系统测试和验收测试，从用户视角验证整个系统的端到端行为。通过分层测试策略，我们能够在保证质量的同时控制测试维护成本。下图展示了本项目的四层测试金字塔结构及其各层之间的数量比例关系。

```
        ┌──────────────────────────┐
        │    验收测试 (UAT)         │  ← 用户场景、满意度
        │    ~5% 数量占比            │
        ├──────────────────────────┤
        │    系统测试 (System)       │  ← 性能、压力、可靠性
        │    ~15% 数量占比           │
        ├──────────────────────────┤
        │    集成测试 (Integration)   │  ← 模块间接口、数据流
        │    ~30% 数量占比           │
        ├──────────────────────────┤
        │    单元测试 (Unit)         │  ← 函数/类级别
        │    ~50% 数量占比           │
        └──────────────────────────┘
```

**测试层级详细定义如下表所示：**

| 层级 | 测试类型 | 测试对象 | 测试工具 | 执行频率 | 预计用例数 |
|------|----------|----------|----------|----------|-----------|
| L1 | 单元测试 | 函数、类、模块 | Unity/PIO native, pytest | 每次提交 | 300+ |
| L2 | 集成测试 | 模块间接口、数据流 | pytest, 自定义脚本 | 每日构建 | 80+ |
| L3 | 系统测试 | 完整系统功能/性能 | 自定义自动化框架 | 每周/里程碑 | 40+ |
| L4 | 验收测试 | 用户场景/满意度 | 手动+问卷 | 每阶段结束 | 20+ |

### 1.2 测试覆盖率目标

测试覆盖率是衡量测试充分性的核心量化指标。本项目设定代码覆盖率目标为 **≥80%**，其中核心模块（传感器驱动、信号处理、AI推理、通信协议）的覆盖率目标更高，为 **≥90%**。覆盖率的计算基于行覆盖率（Line Coverage）和分支覆盖率（Branch Coverage）两个维度，确保代码的关键逻辑路径均被测试用例覆盖。覆盖率数据通过 CI/CD 流水线自动采集，每次提交和合并请求（Pull Request）均会生成覆盖率报告，并设置覆盖率下降的门禁规则，防止新代码引入未覆盖的缺陷。

**各模块覆盖率目标分配如下：**

| 模块 | 语言/框架 | 行覆盖率目标 | 分支覆盖率目标 | 覆盖率工具 | 门禁规则 |
|------|-----------|-------------|---------------|-----------|---------|
| 传感器驱动 (TMAG5273/BNO085) | C/C++ | ≥90% | ≥85% | gcov + lcov | 不得下降 |
| 信号处理 (滤波/特征提取) | C/C++ | ≥95% | ≥90% | gcov + lcov | 不得下降 |
| ESP-NOW通信层 | C/C++ | ≥85% | ≥80% | gcov + lcov | 不得下降 |
| TFLite Micro推理 | C/C++ | ≥80% | ≥75% | gcov + lcov | 允许±2%波动 |
| 上位机数据解析 | Python | ≥90% | ≥85% | coverage.py | 不得下降 |
| AI推理接口 | Python | ≥85% | ≥80% | coverage.py | 允许±3%波动 |
| REST API | Python | ≥90% | ≥85% | coverage.py | 不得下降 |
| 3D前端 (Three.js) | JavaScript | ≥70% | ≥65% | Jest + istanbul | 允许±5%波动 |
| Unity XR (Phase2) | C# | ≥70% | ≥65% | Unity Test Framework | 允许±5%波动 |

### 1.3 测试环境要求

## 2. 固件单元测试

### 2.1 传感器驱动测试

传感器驱动是整个系统的数据源头，其可靠性直接影响后续所有处理环节。TMAG5273 驱动测试需要覆盖 I2C 总线初始化、寄存器读写、多路复用器（TCA9548A）通道切换、磁力计数据读取和数据有效性校验等关键功能。BNO085 驱动测试需要覆盖 SPI/I2C 通信初始化、传感器融合模式配置、四元数/欧拉角/线性加速度数据获取以及传感器校准流程。所有驱动测试均需要在真实硬件和模拟器（Mock I2C/SPI）两种环境下执行，确保硬件相关代码和业务逻辑代码能够被独立验证。测试过程中需要特别关注 I2C 总线时钟拉伸（Clock Stretching）、总线挂死恢复以及多路复用器切换时序等边缘场景。

### 2.2 信号处理测试

信号处理模块负责将原始传感器数据转化为可供AI模型使用的特征向量，是连接硬件层和算法层的关键桥梁。测试需要覆盖以下几个子模块：低通滤波器（巴特沃斯/Chebyshev）、移动平均滤波器、互补滤波器、Kalman滤波器、Madgwick姿态融合算法以及时域/频域特征提取算法。每个滤波算法需要使用已知的合成信号进行验证，确保其频率响应、相位延迟和稳态误差符合设计规格。特征提取测试需要验证计算结果与参考实现（如 Python numpy 实现）的数值一致性，误差应在浮点精度范围内。

### 2.5 测试框架与工具

## 3. 上位机单元测试

### 3.1 数据解析器测试

上位机数据解析器负责接收来自固件的二进制数据流，并将其解析为结构化的传感器数据对象。测试需要覆盖多种数据协议格式（ESP-NOW 原始帧、WiFi SSE 文本流、调试日志格式），验证数据头校验、CRC校验、字节序转换、时间戳同步和多帧拼接等功能。解析器测试还需要验证异常数据的处理能力，包括截断帧、校验失败帧、乱序帧和重复帧等边缘情况。测试用例需要使用预录制的真实数据流（PCAP 或二进制 dump）作为输入，确保解析器与固件的协议实现保持完全一致。

**数据解析器测试用例矩阵：**

| 测试ID | 测试场景 | 输入数据 | 预期输出 | 通过标准 |
|--------|---------|---------|---------|---------|
| SW-PARSE-001 | 标准帧解析 | 完整有效数据帧 | 结构化对象 | 所有字段值正确 |
| SW-PARSE-002 | CRC校验通过 | 有效帧+正确CRC | 解析成功 | 返回数据对象 |
| SW-PARSE-003 | CRC校验失败 | 有效帧+错误CRC | 拒绝该帧 | 抛出CRCError |

## 6. 用户验收测试

### 6.2 测试场景设计
